# Итог проекта Маняша

Дата: 2026-05-01  
Статус: локальный продуктовый RC / demo-ready  
Production-статус: не production-ready без внешнего staging evidence и Postgres Alembic evidence

## Коротко

Маняша — это AI-виджет юридического помощника, который можно встроить на сайт. Он помогает пользователю спокойно описать ситуацию с долгами, банкротством, приставами, доходом, имуществом и коллекторами, а затем переводит разговор в понятный следующий шаг: диагностика, предварительный итог, консультация или передача заявки оператору.

Проект состоит не только из чат-виджета. Внутри есть полный путь:

1. Пользователь общается с Маняшей на сайте.
2. Маняша собирает диагностические данные по ситуации.
3. Пользователь получает короткий предварительный итог.
4. При необходимости пользователь оставляет заявку.
5. Оператор или юрист видит заявку в LeadInbox и Case Workspace.
6. Система помогает оператору понять риск, качество лида, нужные документы, следующий шаг и готовность дела.

## Для чего нужен проект

Проект нужен, чтобы превратить обычный юридический сайт из статичной страницы в понятный интерактивный путь для клиента.

Маняша решает несколько задач:

- отвечает на первые вопросы пользователя простым языком;
- не обещает списание долгов и не заменяет юриста;
- помогает собрать первичную картину ситуации;
- показывает пользователю, что уже понятно и что нужно уточнить;
- формирует предварительный мини-отчёт;
- помогает аккуратно передать заявку юристу;
- даёт оператору структурированное рабочее пространство по заявке.

Главная роль Маняши — не “продать любой ценой”, а снизить тревогу пользователя, собрать полезный контекст и подготовить правильный следующий шаг.

## Какую роль выполняет Маняша

Маняша работает как первый цифровой помощник между пользователем и юридической командой.

Для пользователя она:

- встречает на сайте;
- отвечает на вопросы про долги, приставов, МФЦ, суд, доход, имущество, коллекторов;
- задаёт уточняющие вопросы;
- не переспрашивает то, что уже известно;
- показывает диагностическую карточку;
- формирует предварительный итог;
- предлагает консультацию только когда это логично.

Для оператора или юриста она:

- передаёт заявку с диагностическим контекстом;
- показывает риск и качество лида;
- подсказывает следующий лучший шаг;
- формирует список документов;
- помогает написать follow-up сообщение клиенту;
- показывает readiness gate — готово ли дело к звонку, нужны ли документы или проверка юристом;
- даёт внутреннюю сводку дела для копирования.

## Как построен проект

Проект состоит из нескольких слоёв.

### 1. Frontend-виджет

Виджет находится в `frontend/public/widget` и подключается к странице через frontend shell.

Основные части:

- `widget-core.js` — главный runtime виджета: отправка сообщений, UI, голос, состояние, связь модулей.
- `widget-chat-transport.js` — отправка сообщений в backend и обработка ошибок сети.
- `widget-diagnostics.js` — диагностика ситуации пользователя на frontend.
- `widget-handoff.js` — логика консультации и передачи заявки.
- `widget-demo.js` — demo mode с prepared answers.
- `widget-media.js` — видео/анимационные состояния Маняши.
- `widget-text.js` — обработка текста, `speech_reply`, подготовка текста для озвучки.
- `widget.html` — чистая оболочка portable iframe для встраивания на другие сайты.

### 2. Backend

Backend находится в `app.py`.

Он отвечает за:

- `/api/manyasha/chat` — основной чат Маняши;
- LLM/provider path и fallback;
- relevance guard, чтобы плохой ответ provider не ушёл пользователю;
- intent-aware fallback по юридическим темам;
- `reply` и `speech_reply` contract;
- handoff/consult endpoints;
- LeadInbox API;
- Case Workspace API;
- readiness/health endpoints;
- production env validation;
- portable embed script `/embed.js`.

### 3. Диагностика

Диагностика собирает лёгкое состояние по разговору:

- сумма долга;
- тип долгов;
- приставы;
- официальный доход;
- имущество, квартира, ипотека;
- коллекторы;
- просрочки и исполнительные производства;
- предварительный route hint;
- риск;
- недостающие данные.

Эти данные используются и во frontend, и backend.

### 4. Client report

Пользователь может получить предварительный итог:

- что уже понятно;
- что важно проверить;
- следующий безопасный шаг;
- что подготовить.

Отчёт можно скопировать или явно отправить себе на email. Отправка требует явного действия пользователя и согласия.

### 5. LeadInbox и Case Workspace

Это операторская часть проекта.

LeadInbox позволяет:

- видеть заявки;
- искать и фильтровать;
- менять статус;
- оставлять внутренние заметки;
- видеть contact actions;
- видеть summary по отправленному report email.

Case Workspace показывает:

- контакт и статус;
- диагностику;
- риски;
- quality score;
- next best action;
- список документов;
- follow-up message builder;
- decision checklist;
- readiness gate;
- internal case summary/export.

## Может ли виджет переноситься на другой сайт

Да. Виджет специально подготовлен как portable embed.

Это значит, что внешний сайт не должен копировать внутрь себя весь код Маняши. Сайт подключает один script, а сама Маняша открывается в отдельном iframe. Благодаря iframe стили сайта и стили Маняши не должны конфликтовать.

Есть отдельный dev-стенд для проверки переносимости:

`frontend/public/embed-letaibe-dev.html`

Он имитирует внешний media/news сайт и показывает, что Маняша может быть установлена не только на банкротный лендинг, но и на другой сайт как внешний виджет.

## Что нужно, чтобы поставить Маняшу на другой сайт

### Со стороны владельца сайта

Нужно:

- дать домен сайта;
- разрешить добавить маленький embed-код на страницу;
- разрешить загрузку внешнего script и iframe Маняши;
- если на сайте есть строгая защита, добавить домены Маняши в разрешённые источники;
- не блокировать плавающие виджеты стилями сайта.

Простыми словами: владелец сайта даёт место для вставки и разрешает Маняше загрузиться.

### Со стороны Маняши

Нужно подготовить:

- публичный backend адрес, например `https://api.example.ru`;
- публичный widget host, где лежит `widget.html`;
- `site_key` для конкретного сайта;
- `install_token`, если включена проверка установки;
- домен сайта в allowlist;
- CORS/iframe permissions;
- рабочий AI provider или fallback;
- HTTPS.

Простыми словами: Маняша должна дать сайту готовый код, ключи и рабочий сервер.

## Что нужно, чтобы выложить проект на домен

Для локального demo сейчас используются адреса `127.0.0.1`. Они работают только на компьютере разработчика. Для домена нужны публичные адреса.

Минимально нужно:

1. Домен для сайта, например `manyasha.example.ru`.
2. Хостинг для frontend.
3. Публичный backend/API, например `api.manyasha.example.ru`.
4. HTTPS для сайта и API.
5. PostgreSQL вместо локального SQLite.
6. Redis для production/staging, если требуется конфигурацией.
7. Production/staging env secrets.
8. Настроенный CORS allowlist.
9. Настроенный partner domain allowlist.
10. Проверка `/healthz`, `/readyz`, install-health и отправки сообщения.

Важно: без внешнего staging evidence и Postgres-backed Alembic evidence проект нельзя честно называть production-ready.

## Ограничения и важные правила

- Маняша не должна обещать гарантированное списание долгов.
- Маняша не заменяет юриста.
- Диагностика предварительная и должна проверяться документами.
- Demo mode не должен случайно включаться в normal mode.
- Реальные secrets нельзя хранить в git.
- Email/report flow должен быть только по явному действию пользователя.
- LeadInbox и Case Workspace не должны раскрывать raw chat history, raw report text, install tokens или API keys.

## Итог

Маняша — это не просто чат-бот на сайте. Это связка из пользовательского AI-виджета, диагностики, безопасного handoff и операторского рабочего пространства.

Главная ценность проекта:

- пользователю проще начать разговор о сложной юридической ситуации;
- юрист получает не пустую заявку, а контекст;
- оператор видит, что делать дальше;
- виджет можно переносить на другие сайты через portable embed;
- архитектура уже разделяет demo, normal, widget shell, backend, diagnostics и operator workspace.

Проект готов для локальной демонстрации, продуктового review и подготовки staging. Для реального production нужны домен, публичный backend, база, secrets, staging evidence и Postgres migration evidence.
